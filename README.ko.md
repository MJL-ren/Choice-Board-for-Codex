# Choice Board for Codex

[English](README.md) | **한국어**

> **답변은 더 많이, 타이핑은 더 적게.**
>
> 비공식 커뮤니티 프로젝트입니다. OpenAI와 제휴 관계가 없으며, OpenAI의 보증을 받지 않았습니다.

Choice Board는 서로 연관된 Codex 질문 묶음을 하나의 작은 인터랙티브 보드로
바꿔 줍니다. 단일 선택이나 다중 선택 옵션을 고르고, 자유 입력이나 짧은 메모를
덧붙이고, 뜻이 불분명한 질문에는 설명을 요청한 뒤, 완성된 답변을 같은 대화로
그대로 돌려보낼 수 있습니다.

MCP 서버도, localhost 서비스도, 터널도, 데이터베이스도, 별도의 웹 앱도 전혀
필요하지 않습니다.

## 만들게 된 이유

다른 도구들의 선택지 UI에 익숙해져 있다 보니, 평소 Codex 작업에서도 그 리듬이
아쉬웠습니다. Codex에 내장된 것 중 가장 비슷한 흐름은 Plan 모드에 묶여
있었고요. 일반 작업 중에 서로 연관된 질문이 대여섯 개씩 나오면, 여전히 답을
하나하나 타이핑하면서 번호를 맞추고 빠뜨린 것이 없는지 확인해야 했습니다.
그때마다 흐름이 끊겨서, 결국 제가 쓰고 싶었던 인터랙션을 직접 만들게
되었습니다.

질문이 몇 개 되지 않을 때는 모든 것이 한 보드에 모여 있어 선택지를 나란히
비교하기 좋습니다. 질문이 많아지면 뒤로 가기, 건너뛰기, 메모, 최종 검토가
있는 안내형 흐름으로 바뀝니다. 제가 써 본 다른 선택지 인터페이스에서는 보지
못했던 기능도 하나 더했습니다. 바로 제한적 분기(bounded branching)입니다.
앞선 답변 하나로 더 이상 해당되지 않는 미리 작성된 후속 질문을 숨길 수
있어서, Codex가 불필요한 질문과 대화 턴을 줄이면서도 더 복잡한 맥락을 모을 수
있습니다.

보드는 Codex 자체의 테마와 네이티브 컨트롤을 그대로 따릅니다. 단순한 폼을
굼뜨게 만들 수 있는 장식용 CSS 효과나 무거운 애니메이션은 일부러 넣지
않았습니다. 사양이 넉넉하지 않은 하드웨어에서는 특히 그렇습니다. 그리고 AI가
만든 선택지가 전문 용어나 숨은 맥락에 기대고 있을 때는, 뜻을 짐작하는 대신 그
질문에 설명 요청 표시를 남길 수 있습니다.

작은 비공식 도구이지만, 언젠가 이런 다중 질문 인터랙션이 Codex의 기본 기능이
되는 모습을 볼 수 있다면 기쁠 것 같습니다.

## 실제 동작 모습

### 짧은 보드는 선택지를 한곳에 모아 줍니다

질문 수가 적으면 선택지가 한 보드에 그대로 머뭅니다. 답을 고르고, 메모를
덧붙이고, 설명을 요청하고, 번호를 매긴 답장을 타이핑할 필요 없이 한 번에
전송할 수 있습니다.

![답변이 선택되고 답변 메모 입력란이 열려 있는 간결형 Choice Board](assets/screenshots/choice-board-answer-note-dark.png)

### 긴 보드는 선택한 경로를 따라갑니다

분기형 보드는 경로를 정하는 선택 하나로 시작합니다. 그 답변에 여전히 해당되는
미리 작성된 후속 질문만 표시됩니다.

![Mix both가 선택된 분기형 Choice Board의 첫 단계](assets/screenshots/branching-route-mixed-dark.png)

다음 단계는 경로에 따라 달라지지만, 뒤로 가기·건너뛰기·다음 같은 익숙한
컨트롤은 그대로 유지됩니다.

![야외 활동 다중 선택과 안내형 내비게이션이 있는 이후 분기 단계](assets/screenshots/branching-outdoor-multi-select-dark.png)

### 설명은 먼저, 검토는 한 번에

질문이 명확하지 않으면 그 자리에서 바로 설명을 요청할 수도 있고, 나중에 볼 수
있게 표시해 두고 나머지를 먼저 끝낼 수도 있습니다.

![즉시 또는 지연 설명 요청을 제안하는 안내형 Choice Board](assets/screenshots/branching-explanation-request-dark.png)

무언가를 전송하기 전에, 보드가 활성 답변과 아직 해결되지 않은 설명 요청을
하나의 검토 화면으로 모아 줍니다.

![전송 준비된 설명 요청이 포함된 최종 Choice Board 검토 화면](assets/screenshots/branching-review-dark.png)

## 인터랙션 모드

| 모드 | 적합한 경우 | 동작 |
| --- | --- | --- |
| 간결형(Compact) | 고정 질문 1–3개 | 질문을 한 화면에 함께 보여 주고 한 번에 제출합니다. |
| 안내형(Guided) | 고정 질문 4개 이상 | 질문을 하나씩 보여 주며 뒤로 가기, 건너뛰기, 설명 요청, 메모, 최종 검토를 제공합니다. |
| 제한적 분기(Bounded branching) | 앞선 선택 하나에 실제로 의존하는 미리 작성된 후속 질문 | `show_if` 조건을 한 층만 사용하고, 숨겨진 상태를 비우며, 실제로 지나온 경로만 정확히 반환합니다. |

안내형과 제한적 분기 보드에 임의로 정한 질문 수 상한은 없습니다. 다만
렌더러는 필드별 한도와 프래그먼트 크기 안전 한도를 계속 적용합니다. 중첩
조건식과 모델이 즉석에서 생성하는 후속 분기는 의도적으로 지원 범위에서
제외했습니다.

## 요구 사항과 지원 현황

- Codex가 포함된 ChatGPT 데스크톱 앱
- OpenAI의 **Visualize** 플러그인 설치 및 활성화
- Codex가 사용할 수 있는 Python 3.10 이상

| 환경 | 상태 |
| --- | --- |
| Windows용 Codex Desktop | 실사용 테스트 완료 |
| macOS용 Codex Desktop | 지원 대상이지만 아직 별도 검증 전 |
| 모바일 | 현재 인터랙티브 미지원, 일반 텍스트 폴백만 제공 |
| Codex CLI 및 IDE 확장 | 현재 지원을 주장하지 않음 |

플러그인 사용 가능 여부는 요금제, 워크스페이스 설정, 역할, 사용 환경, 지역에
따라 다를 수 있습니다. Visualize를 사용할 수 없거나 휴대폰 화면에 인라인
시각화 지시문이 그대로 노출되는 경우, 스킬은 같은 대화 안에서 곧바로 동등한
번호 매김 텍스트 질문으로 전환됩니다.

OpenAI는 현재 Visualizations가 대상 모바일 계정에 순차 적용되는 중이라고
안내하고 있습니다. 다만 그 배포가 Choice Board와의 호환성이나 지원 시점을
보장하지는 않습니다. 이 프로젝트는 대상 모바일 계정에서 렌더링, 인터랙션,
후속 메시지 전달, 재시도까지 검증되기 전까지 텍스트 폴백을 유지합니다.
자세한 내용은 [공식 Visualizations 문서](https://learn.chatgpt.com/docs/visualizations)를
참고하세요.

기본 인터페이스 문구는 영어와 한국어를 지원합니다. 질문과 선택지 텍스트는
호출한 쪽이 사용한 언어를 그대로 따르며, 지원되지 않는 인터페이스 로캘은
영어로 대체됩니다.

## 설치

1. ChatGPT 데스크톱 앱에서 **Plugins**를 열고, OpenAI의 **Visualize**를 찾아
   설치하거나 활성화합니다.
2. Codex에게 이 저장소의 스킬 설치를 요청합니다:

   ```text
   $skill-installer Install the skill from https://github.com/MJL-ren/Choice-Board-for-Codex/tree/main/skills/codex-choice-board
   ```

3. 슬래시 메뉴를 열어 `/codex-choice-board`를 실행해 봅니다. 바로 나타나지
   않으면 새 Codex 작업을 시작하거나 데스크톱 앱을 재시작한 뒤, 스킬과
   Visualize가 모두 활성화되어 있는지 확인하세요.

OpenAI는 스킬 직접 설치를 로컬 제작·실험용 경로로 안내하고 있습니다. 더 넓은
재사용 배포에는 패키징된 Codex 플러그인이 권장 경로이며, 이 저장소는 그
패키징 경로를 검토하는 동안 우선 스킬 폴더를 직접 공개하고 있습니다.

## 사용해 보기

항상 동작이 보장되는 호출 형태는 다음과 같습니다:

```text
$codex-choice-board 주말 활동을 정할 수 있게 질문 6개를 해 줘. 선호는 여러 개 고를 수 있게 하고, 답변에 메모도 남기고, 결정하기 전에 설명도 요청할 수 있게 해 줘.
```

"선택지를 줘"나 "이거 초이스 보드로 보여 줘" 같은 직접적인 요청도 인식됩니다.
기본 동작은 fail-closed로 유지됩니다. 사용자가 활성화 모드를 `suggest`나
`auto`로 직접 바꾸지 않는 한, 스킬은 무관한 요청에 보드를 제안하거나 열지
않습니다.

바로 렌더링해 볼 수 있는 예제도 준비되어 있습니다:

- [`examples/basic-en.json`](examples/basic-en.json)
- [`examples/basic-ko.json`](examples/basic-ko.json)
- [`examples/guided-ko.json`](examples/guided-ko.json)
- [`examples/branching-ko.json`](examples/branching-ko.json)

## 동작 방식

1. Codex가 질문들을 표준(canonical) Choice Board 스키마로 변환합니다.
2. 결정론적 Python 렌더러가 중복 JSON 키, 유한하지 않은 숫자, 알 수 없는
   필드, 유효하지 않은 스키마 값을 엄격히 거부한 뒤, 표준 데이터를 루트
   범위로 고정된 HTML 프래그먼트에 주입합니다.
3. Visualize가 호스트 테마를 따르는 네이티브 컨트롤로 보드를 표시합니다.
4. 보드가 `window.openai.sendFollowUpMessage(...)`를 호출해 현재 대화에 후속
   메시지 하나를 요청합니다.
5. 응답을 받는 Codex 작업은 응답을 사용하기 전에, 번들로 제공되는
   봉투(envelope) 검증기를 같은 표준 명세에 대해 실행합니다. 검증기는 마커,
   읽기용 요약과의 일치, 폼 식별자, 정확한 질문 키, 답변 타입, 선택지 값,
   흐름 다이제스트, 활성 경로, 완료 부모(completion parent), 중복 제출
   식별자를 확인합니다.

호스트 호출이 정상 완료되었다는 사실을 메시지가 실제로 대화에 도착했다는
증거로 간주하지 않습니다. 보드는 초안을 계속 유지하며, 같은 `submission_id`로
바이트 단위까지 동일한 명시적 재시도를 제공합니다. 자동 재시도는 절대
사용하지 않습니다.

## 데이터와 안전 경계

- 제출된 답변은 현재 진행 중인 Codex 대화에 표시됩니다.
- 스킬은 답변을 별도의 서버나 데이터베이스로 전송하지 않습니다.
- 표준 JSON과 렌더링된 HTML은 해당 작업의 로컬 시각화 디렉터리에 생성됩니다.
  이 디렉터리의 보존 정책은 Codex Desktop이 관리합니다. 제출이나 앱 재시작
  후에도 파일이 남아 있을 수 있고, 복원된 보드에는 `initial_*` 답변 상태가
  포함되어 있을 수 있습니다. Choice Board에는 호스트 정리 API가 없으며, 자동
  삭제를 약속하지 않습니다.
- 반환된 메시지는 결정론적 봉투 검증을 위해 같은 작업 디렉터리에 잠시 복사될
  수 있습니다. 원래의 작업, 재시도, 설명, 완료 흐름에 더 이상 필요하지
  않게 되면 그 추가 사본은 삭제해야 합니다. 생성된 보드, 검증용 사본, 사용자
  응답은 이 저장소에 포함하지 않습니다.
- 스킬이 로컬에 저장하는 것은 사용자가 직접 제어하는 활성화 설정뿐입니다.
- 비밀 정보, 자격 증명, 민감한 개인정보, 또는 파괴적이거나 외부에 영향을
  주는 작업의 최종 승인을 수집하는 용도로 보드를 사용하지 마세요.
- 사람이 읽는 라벨은 표시용일 뿐입니다. 권위는 압축된 JSON 봉투에 있으며,
  둘이 어긋나면 안전하게 실패(fail-closed)합니다.

신고 방법과 신뢰 경계에 대한 자세한 내용은 [`SECURITY.md`](SECURITY.md)를
참고하세요.

## 실사용으로 검증했습니다

Choice Board는 정적 목업 수준을 넘어 테스트되었습니다. 자동화된 테스트
스위트는 스키마 검증, 이스케이프 처리, 컴파일, 분기 상태, 재시도 동작,
반환 메시지 전체를 다룹니다. 브라우저 테스트는 간결형, 안내형, 답변 메모,
분기, 로캘 폴백, 30문항 안내형 흐름, 320px과 736px 레이아웃, Codex
라이트·다크 테마를 확인합니다. 실제 Windows Codex Desktop 환경에서도 제출,
취소 후 복구, 뒤로 가기 상태 보존, 즉시·지연 설명 요청, 답변 메모, 제한적
분기 하나까지 직접 실행해 확인했습니다.

이 테스트들이 스크린 리더 인증, 모바일 기기 자동 감지, 검증되지 않은 Codex
환경에 대한 지원을 주장하는 것은 아닙니다. 정확한 검증 범위는
[`docs/TESTING.md`](docs/TESTING.md)에 기록되어 있습니다.

## 개발

브라우저 테스트 의존성 설치:

```powershell
npm install
npx playwright install chromium
```

Python 테스트 스위트 실행:

```powershell
python -m unittest discover -s tests -p "test_*.py" -v
```

스킬 패키지 검증:

```powershell
python -m pip install PyYAML==6.0.2
python tests/run_official_skill_validator.py
```

이 실행 스크립트는 고정된 공식 커밋에서 OpenAI `skill-creator` 빠른 검증기를
내려받고, 실행 전에 SHA-256 해시를 확인합니다. 브라우저 명령과 생성
픽스처(fixture) 단계는 [`docs/TESTING.md`](docs/TESTING.md)에 문서화되어
있습니다.

Playwright 설치 후 전체 브라우저 회귀 테스트 실행:

```powershell
npm run test:browser
```

적대적 테스트 픽스처, 브라우저 테스트로 바꾸는 과정에는 Codex와 GPT-5.6의
도움을 받았습니다. 반면 제품의 경계는 사람이 의도적으로 내린 선택입니다.
데스크톱 우선 인터랙션, 일반 텍스트 폴백, 서버 없음, 기본값으로서의 명시적
활성화, 답변 메모, 설명 요청, 그리고 단 한 층의 제한적 분기까지 모두요.
